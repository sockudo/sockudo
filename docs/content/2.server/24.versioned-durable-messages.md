## Overview

Versioned durable messages (also called mutable messages) let you publish a message once, then update, delete, or append to it over time. Every mutation is stored as a new version. Subscribed V2 clients receive mutation events in real time. The HTTP API gives server-side callers read access to the current state and version history.

This is a **Protocol V2 only** feature. V1 (Pusher-compatible) clients never receive mutation envelopes and have no compatible HTTP surface.

Use cases:

- Live document editing where each edit replaces prior state
- Incrementally-built LLM output streamed token by token via append
- Soft-deletable chat messages with client-side rendering
- Live scoreboards or price feeds with in-place replacement

Requirements:

- `versioned_messages.enabled = true`
- Durable channel history enabled for the target channel
- V2 clients (`?protocol=2`) to receive mutation events in real time

## Configuration

```toml [config/config.toml]
[versioned_messages]
enabled = true
driver = "memory"
max_page_size = 100
retention_window_seconds = 0
purge_interval_seconds = 300
purge_batch_size = 1000
max_purge_per_tick = 100000
```

Or via environment variables:

```bash
VERSIONED_MESSAGES_ENABLED=true
VERSIONED_MESSAGES_DRIVER=postgres
VERSIONED_MESSAGES_MAX_PAGE_SIZE=100
VERSIONED_MESSAGES_RETENTION_WINDOW_SECONDS=0
```

| Field | Type | Default | Description |
|---|---|---|---|
| `enabled` | `bool` | `false` | Enable versioned message storage and HTTP surfaces |
| `driver` | `string` | `memory` | Storage backend: `memory`, `postgres`, `mysql`, `dynamodb`, `scylladb`, `surrealdb` |
| `max_page_size` | `usize` | `100` | Maximum items returned per page on version-history queries |
| `retention_window_seconds` | `u64` | `0` | How long to retain version entries in seconds. `0` disables expiry |
| `purge_interval_seconds` | `u64` | `300` | Interval between background purge worker runs (for backends without native TTL) |
| `purge_batch_size` | `usize` | `1000` | Maximum rows deleted per purge query |
| `max_purge_per_tick` | `usize` | `100000` | Hard cap on rows deleted per purge tick |

Backends with native row TTL (ScyllaDB `USING TTL`, DynamoDB TTL attribute) apply `retention_window_seconds` at the storage layer. Other backends (MySQL, PostgreSQL, SurrealDB, Memory) enforce it via the background purge worker.

## Message Lifecycle

Every mutable message starts with a `message.create` action and is identified by its `messageSerial`. Subsequent mutations reference the same `messageSerial`. Each mutation creates a new `versionSerial`.

```
message.create  (messageSerial = "msg-123", versionSerial = "v1", data = "Hello")
      ↓
message.update  (messageSerial = "msg-123", versionSerial = "v2", data = "Hello, world")
      ↓
message.append  (messageSerial = "msg-123", versionSerial = "v3", data = " How are you?")
      ↓
message.delete  (messageSerial = "msg-123", versionSerial = "v4")
```

The `messageSerial` is the stable identity of the logical message throughout its life. The `versionSerial` identifies a specific mutation version.

### Actions

| Action | Effect | Data field |
|---|---|---|
| `message.create` | Establishes the initial message | Full initial payload |
| `message.update` | Replaces the message payload (shallow mixin) | Replacement fields |
| `message.delete` | Soft-deletes — a new version marks the message as deleted | Optional tombstone payload |
| `message.append` | Appends a string fragment and stores the full rolled-up result | String fragment |

`message.delete` is always a soft delete. The version history is preserved and queryable. The latest-view endpoint reflects the deleted state.

`message.append` is string-only. The stored and broadcast version contains the full rolled-up string, not only the fragment.

## Version Metadata Headers

Mutation events delivered to V2 clients include version metadata in `extras.headers`:

| Header | Type | Description |
|---|---|---|
| `sockudo_action` | `string` | One of `message.create`, `message.update`, `message.delete`, `message.append` |
| `sockudo_message_serial` | `string` | Stable identity of the logical message |
| `sockudo_version_serial` | `string` | Identity of this specific mutation version |
| `sockudo_history_serial` | `number` | Channel history serial for this version event |
| `sockudo_version_timestamp_ms` | `number` | Unix millisecond timestamp of this version |

A V2 client receives these as part of the event envelope:

```json
{
  "event": "my-event",
  "channel": "docs:readme",
  "data": "Hello, world",
  "serial": 12,
  "stream_id": "01HSZ7J0J4P8S7X3JCM4H9G2A1",
  "message_id": "b2fd1961-78db-4d49-bf59-02396c67948f",
  "extras": {
    "headers": {
      "sockudo_action": "message.update",
      "sockudo_message_serial": "msg-123",
      "sockudo_version_serial": "v2",
      "sockudo_history_serial": 12,
      "sockudo_version_timestamp_ms": 1775292000123
    }
  }
}
```

V1 clients never receive these headers and never receive `message.update`, `message.delete`, or `message.append` events.

## Server-Side HTTP API

All mutation endpoints require:

- Signed query authentication (same as all other `/apps/*` endpoints)
- `versioned_messages.enabled = true`
- Durable history enabled for the target channel
- Returns `422` when either requirement is not met

`messageSerial` and `versionSerial` values are limited to 128 bytes.

### Create a Message

Creating a versioned message uses the standard event publish endpoint. When durable history is enabled and `versioned_messages.enabled = true`, the first publish for a given event is tracked as `message.create`.

`POST /apps/{appId}/events`

A regular trigger is treated as the initial creation. No special body field is required.

::code-group

```ts [Node.js]
await client.trigger("docs:readme", "my-event", "Hello, world");
```

### Get Latest Visible Message

`GET /apps/{appId}/channels/{channelName}/messages/{messageSerial}`

Returns the current winning visible version for one logical message.

**Response** `200 OK`

```json
{
  "channel": "docs:readme",
  "item": {
    "message_serial": "msg-123",
    "action": "message.update",
    "version_serial": "v2",
    "event_name": "my-event",
    "data": "Hello, world",
    "published_at_ms": 1775292000123
  }
}
```

Returns `404` when the `messageSerial` does not exist. Returns `422` when versioned messages are disabled.

::code-group

```ts [Node.js]
const result = await client.getMessage("docs:readme", "msg-123");
console.log(result.item.data);
```

```python [Python]
result = client.get_message("docs:readme", "msg-123")
print(result.item)
```

```ruby [Ruby]
result = client.get_message("docs:readme", "msg-123")
puts result
```

```go [Go]
result, err := client.GetMessage("docs:readme", "msg-123")
```

```php [PHP]
$result = $client->getMessage("docs:readme", "msg-123");
```

```csharp [.NET]
var result = await client.GetMessageAsync<object>("docs:readme", "msg-123");
```

### Get Message Versions

`GET /apps/{appId}/channels/{channelName}/messages/{messageSerial}/versions`

Returns preserved versions in deterministic `version_serial` order. Losing concurrent versions remain queryable.

**Query parameters**

| Parameter | Description |
|---|---|
| `limit` | Page size, capped at `max_page_size` |
| `direction` | `newest_first` or `oldest_first` |
| `cursor` | Opaque cursor from a previous page |

**Response** `200 OK`

```json
{
  "channel": "docs:readme",
  "direction": "newest_first",
  "limit": 100,
  "has_more": false,
  "next_cursor": null,
  "items": [
    {
      "message_serial": "msg-123",
      "action": "message.update",
      "version_serial": "v2",
      "event_name": "my-event",
      "data": "Hello, world",
      "published_at_ms": 1775292001000
    },
    {
      "message_serial": "msg-123",
      "action": "message.create",
      "version_serial": "v1",
      "event_name": "my-event",
      "data": "Hello",
      "published_at_ms": 1775292000000
    }
  ]
}
```

::code-group

```ts [Node.js]
const page = await client.getMessageVersions("docs:readme", "msg-123", {
  limit: 50,
  direction: "newest_first",
});
for (const version of page.items) {
  console.log(version.action, version.data);
}
```


### Update a Message

`POST /apps/{appId}/channels/{channelName}/messages/{messageSerial}/update`

Applies a shallow-mixin update to a mutable message. Subscribed V2 clients receive a `sockudo:message.update` event.

**Request fields**

| Field | Description |
|---|---|
| `name` | Replacement event name |
| `data` | Replacement data payload |
| `extras` | Replacement extras |
| `clear_fields` | List of fields to clear: `name`, `data`, `extras` |
| `client_id` | Identity of the actor performing the update |
| `socket_id` | Signed-in V2 socket for capability-bound authorization |
| `description` | Human-readable reason for the update |
| `metadata` | Arbitrary operator metadata |

Validation rules:

- At least one of `name`, `data`, `extras`, or `clear_fields` must be present
- `clear_fields` must not contain duplicates
- A field cannot appear in both a replacement and `clear_fields` in the same request

**Response** `200 OK`

```json
{
  "channel": "docs:readme",
  "message_serial": "msg-123",
  "action": "message.update",
  "accepted": true,
  "status": "applied",
  "version_serial": "v2"
}
```

::code-group

```ts [Node.js]
const result = await client.updateMessage("docs:readme", "msg-123", {
  data: "Hello, world",
  description: "Fix typo",
});
console.log(result.version_serial);
```

```python [Python]

```


### Append to a Message

`POST /apps/{appId}/channels/{channelName}/messages/{messageSerial}/append`

Appends a string fragment to the current message state and stores the full rolled-up result. Subscribed V2 clients receive a `sockudo:message.append` event with the fragment as `data`. The stored version contains the full concatenated string.

**Request fields**

| Field | Required | Description |
|---|---|---|
| `data` | Yes | Non-empty string fragment to append |
| `socket_id` | No | Signed-in V2 socket for capability-bound authorization |
| `client_id` | No | Identity of the actor |
| `description` | No | Human-readable reason |
| `metadata` | No | Arbitrary operator metadata |

Append is string-only. If the current state is not a string, the request is rejected.

::code-group

```ts [Node.js]
await client.appendMessage("docs:readme", "msg-123", {
  data: " How are you?",
});
```

```python [Python]
client.append_message("docs:readme", "msg-123", " How are you?")
```

```ruby [Ruby]
client.append_message("docs:readme", "msg-123", { data: " How are you?" })
```

```go [Go]
client.AppendMessage("docs:readme", "msg-123", map[string]interface{}{
    "data": " How are you?",
})
```

```php [PHP]
$client->appendMessage("docs:readme", "msg-123", ["data" => " How are you?"]);
```

```csharp [.NET]
await client.AppendMessageAsync<object>("docs:readme", "msg-123", new { data = " How are you?" });
```

## Authorization Model

When `socket_id` is omitted on a mutation request, the app-key-authenticated HTTP caller is treated as a privileged server actor. Mutations proceed without capability checks.

When `socket_id` is present, Sockudo resolves the signed-in V2 connection and enforces capability-based authorization:

- `message_update_own` / `message_delete_own` / `message_append_own` — actor must match the original creator `client_id`
- `message_update_any` / `message_delete_any` / `message_append_any` — mutation allowed regardless of original creator identity on matching channels

A `client_id` in the request body, when present alongside `socket_id`, must match the authenticated socket identity.

## Realtime Delivery

V2 clients subscribed to the channel receive mutation events in real time:

| HTTP action | Client event |
|---|---|
| `update` | `sockudo:message.update` |
| `delete` | `sockudo:message.delete` |
| `append` | `sockudo:message.append` |

The initial publish is delivered as a regular event. For the initial publish to be tracked as `message.create`, the channel must have durable history and `versioned_messages.enabled = true`.

V1 clients receive live events for the initial publish only. They never receive `sockudo:message.update`, `sockudo:message.delete`, or `sockudo:message.append` envelopes.

## History Integration

When `versioned_messages.enabled = true`, the channel history endpoint (`GET /apps/{appId}/channels/{channelName}/history`) returns a **latest-visible projection**. History items that belong to a mutable message chain reflect the winning version's payload and action metadata rather than the raw stored payload.

See [Channel History](/server/channel-history#operator-notes-for-mutable-messages) for details on history substitution and related metrics.

## Error Responses

| Status | Reason |
|---|---|
| `400` | Invalid request body or query parameters |
| `404` | `messageSerial` does not exist in the channel |
| `422` | `versioned_messages.enabled` is `false` or durable history is disabled by policy |

## Metrics

- `versioned_message_mutations_total{action,result}` — mutation operations by action and outcome
- `versioned_message_retrieval_total{surface="latest",result}` — latest-view lookups
- `versioned_message_retrieval_total{surface="versions",result}` — version-history lookups
- `versioned_history_substitution_total{result="applied"}` — history items successfully substituted
- `versioned_history_substitution_total{result="missing_latest"}` — substitution failures (operator-visible warning)
