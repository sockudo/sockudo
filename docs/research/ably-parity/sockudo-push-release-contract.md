# Release 4.5: Push Notification Platform - Binding Contract

## Status

Binding pre-implementation contract.

No release 4.5 push implementation code should land unless it is consistent
with this document and the architecture ADR for this release.

## Purpose

Release 4.5 adds an optional, horizontally scalable push notification platform
to Sockudo. It provides device activation, device and channel subscription
registry, direct and channel-based push publishing, asynchronous fanout,
scheduled delivery, cancellation, status tracking, provider feedback handling,
and operator controls for reliability and scale.

The release must remain optional. Existing V1 Pusher-compatible behavior and V2
real-time messaging must continue to compile and run without push dependencies
unless the `push` feature is enabled.

## Goals

Release 4.5 must deliver all of the following:

- Ably-style activation, direct publish, batch publish, async publish, channel
  push, scheduled publish, cancellation, and status tracking.
- Centrifugo-inspired operator controls for configuration, reliability mode,
  dry-run, stale device cleanup, templates, localization, analytics, metrics,
  and deployment modes.
- Provider support for FCM, APNs, Web Push, HMS, and WNS.
- Storage support for memory, PostgreSQL, MySQL, DynamoDB, SurrealDB, and
  ScyllaDB.
- Queue support for memory, Redis, Redis Cluster, NATS JetStream, Pulsar,
  RabbitMQ, Google Pub/Sub, Kafka, Iggy, SQS, and SNS, with documented scale
  suitability.
- Feature-gated provider and backend dependencies.
- A payload transformation layer that produces provider-specific payloads from
  Sockudo push requests, templates, localization, channel context, and device
  attributes.
- Per-device state that is queryable and observable.
- Tenant fairness, provider backpressure, duplicate suppression, and durable
  retry state for reliability mode.

## Non-Goals

Release 4.5 does not include:

- Dashboard UI.
- Client SDK implementation beyond HTTP/protocol documentation and examples.
- A custom device transport replacing FCM, APNs, Web Push, HMS, or WNS.
- End-to-end device delivery guarantees beyond provider acknowledgements.
- Claims that Sockudo can force providers to exceed provider quotas.
- Push support in minimal builds that do not enable the `push` feature.

## V1 And V2 Boundary

Push publishing and activation are HTTP/API surfaces, not Pusher V1 wire
extensions.

Binding rules:

- V1 WebSocket message envelopes must not gain push-only fields.
- V1 channel events may trigger push only through server-side `extras.push`
  processing in HTTP publish paths or internal publish paths that already have
  application authorization.
- V2 can expose richer channel push metadata, but provider payloads must be
  generated by the push transformation layer, not leaked into V1 payloads.
- Push status, activation, cancellation, registry, and admin APIs are HTTP
  surfaces protected by app credentials and explicit push capabilities.

## Ably Parity Surface

### Activation

Release 4.5 supports two activation modes:

- Direct activation: trusted application servers register device identities,
  provider tokens/subscriptions, capabilities, and optional channel push
  subscriptions directly.
- Server-assisted activation: clients request activation material from Sockudo;
  Sockudo returns activation instructions and records the resulting device
  registration after authenticated confirmation.

Activation must record:

- `app_id`
- `device_id`
- provider
- provider token or Web Push subscription endpoint/key material
- token hash
- platform metadata
- channel/topic subscriptions
- locale and timezone hints
- last seen and last confirmed timestamps
- stale/delete eligibility
- provider credential version used for last successful send

### Registry

The registry is authoritative for devices, provider endpoints, channel push
subscriptions, per-device state, and stale cleanup decisions.

List endpoints must use cursor pagination. Offset-only pagination is not
acceptable for any registry list or status list endpoint.

### Direct Publish

Direct publish sends to explicit devices, provider tokens, user IDs, or stored
device selectors. Synchronous direct publish may return per-recipient immediate
validation errors, but large fanout must use async publish.

### Channel Push

Channel push sends to devices subscribed to a Sockudo channel or push topic.
Channel publish may include `extras.push`; push delivery is planned from stored
subscriptions and transformed payloads.

Binding rules:

- Channel push subscription state is part of the push registry.
- Channel fanout planning must be sharded and resumable.
- `extras.push` is input to the transformation layer, not a raw provider JSON
  pass-through.

### Batch Publish

Batch publish accepts multiple direct or channel push items in one request.
Batch results must preserve caller item order and include item-level validation
or `publish_id` references.

### Async Publish

Async publish returns:

```text
202 Accepted
publish_id: <opaque id>
```

The `publish_id` is the canonical handle for status, cancellation, duplicate
suppression, and audit lookup.

### Scheduled Delivery

Scheduled delivery stores a not-before timestamp and optional expiry. It must be
cancellable before dispatch. Broker delay features may optimize scheduling, but
storage is the source of truth.

### Cancellation

Cancellation must support:

- scheduled publishes not yet dispatched
- planned fanout batches not yet enqueued
- queued dispatch jobs not yet delivered to providers where backend semantics
  allow cancellation

Cancellation after provider handoff is best-effort and must be reflected in
status.

### Status

Status tracking must expose:

- publish acceptance state
- planning state
- scheduled/cancelled/expired state
- dispatch progress
- provider acknowledgement and provider error classes
- per-device state
- retry and dead-letter state
- aggregate counts by provider and status

All status list endpoints require cursor pagination.

## Operator Controls

### Configuration

Push configuration must be explicit and feature-gated. Operators must be able to
configure:

- enabled providers
- provider credentials and credential versions
- storage backend
- queue backend per pipeline stage
- worker concurrency
- per-tenant quotas
- provider rate limits
- retry policies
- stale cleanup windows
- reliability mode
- dry-run mode
- template and localization defaults
- analytics retention

### Reliability Mode

Reliability mode persists publish acceptance, fanout plans, dispatch attempts,
provider responses, retry state, and dead letters before returning success for
async acceptance. Memory storage and memory queue are not valid for production
reliability mode.

### Dry Run

Dry-run mode validates auth, payload transformation, targeting, quotas, and
fanout planning without provider dispatch. It must emit status and metrics with
a dry-run dimension.

### Stale Cleanup

Stale device cleanup must be configurable per provider and app. Provider
feedback can mark tokens invalid immediately, while age-based cleanup must use
configurable thresholds and safe batching.

### Templating And Localization

Payload transformation must support named templates, parameter interpolation,
locale selection, fallback locale, provider-specific fields, and validation
before provider dispatch. Raw provider JSON pass-through is not sufficient for
release 4.5.

### Analytics

Analytics must be bounded by retention and must not require scanning unbounded
raw delivery rows for normal dashboards or reports.

### Worker Metrics

Worker metrics must cover accept, plan, schedule, dispatch, provider feedback,
retry, DLQ, stale cleanup, quota, backpressure, and dry-run paths.

## Provider Scope

Release 4.5 provider implementations:

- FCM via service account JSON / OAuth2 and HTTP v1.
- APNs via `.p12` or PEM credentials, with token auth supported where the
  adopted crate and wrapper provide stronger operational behavior.
- Web Push via VAPID and RFC 8291 payload encryption.
- HMS via app credentials and HTTPS provider API.
- WNS via Package SID/client secret or Windows App SDK credentials and HTTPS
  channel URI sends.

Provider wrappers must normalize responses into Sockudo-owned error classes:

- accepted
- retryable provider throttling
- retryable provider outage
- invalid token/subscription
- invalid request
- auth/credential failure
- provider quota/capacity limit
- unknown provider error

## Storage Scope

Supported storage backends:

- memory: tests and development only
- PostgreSQL: small/regional production
- MySQL: small/regional production
- DynamoDB: cloud production and hyperscale candidate
- SurrealDB: supported where feasible, not hyperscale recommended without proof
- ScyllaDB: hyperscale recommended candidate

Storage must cover:

- device registry
- provider endpoint/token records
- channel push subscriptions
- publish records
- scheduled publish records
- fanout plan shards
- dispatch attempts
- provider feedback
- dead letters
- analytics rollups
- credential metadata and encrypted credential material when stored by Sockudo

## Queue Scope

Supported queue backends:

- memory: tests and development only
- Redis: small production, not reliability mode without safer retry semantics
- Redis Cluster: small/regional production, not hyperscale dispatch recommended
- NATS JetStream: regional production and hyperscale candidate with configured
  limits
- Pulsar: hyperscale recommended candidate
- RabbitMQ: regional production
- Google Pub/Sub: cloud production and hyperscale recommended on GCP
- Kafka: hyperscale recommended after partition configuration
- Iggy: hyperscale candidate after benchmark proof
- SQS: cloud production, not 100M fanout planning without aggressive sharding
- SNS: publish fanout only; not a complete worker queue because consumption is
  external

Queue semantics must be documented per stage: accept, plan, schedule, dispatch,
feedback, cleanup, and DLQ.

## Public API Contract

Release 4.5 must expose HTTP APIs for:

- activation
- activation confirmation
- device upsert/delete/get/list
- channel push subscription upsert/delete/list
- direct publish
- batch publish
- async publish
- channel push publish
- scheduled publish
- cancellation
- publish status get/list
- per-device status get/list
- dead-letter list/replay/delete
- provider credential version list/rotate
- stale cleanup trigger/status

Auth rules:

- Existing app key/secret auth may authenticate application-scoped push APIs.
- Admin APIs require explicit server/admin credentials or capability gates.
- Capability checks must separate device registry mutation, publish, status
  read, cancellation, credential management, DLQ replay, and cleanup.
- Request signing must bind body, timestamp, app, and route for sensitive admin
  operations.

All list endpoints must use opaque cursor pagination.

## Internal Abstractions

Required internal traits or equivalent module boundaries:

- `PushStore`: registry, publish records, schedule records, status, attempts,
  dead letters, analytics rollups, credential metadata.
- `PushQueue`: enqueue/dequeue/ack/nack/retry/dead-letter semantics normalized
  across supported queue backends.
- `PushProvider`: provider-specific validation, send, batch send where supported,
  response parsing, credential reload.
- `PushPlanner`: target resolution, channel fanout sharding, quota admission,
  duplicate suppression, schedule-to-dispatch transition.
- `PushWorker`: accept, plan, schedule, dispatch, feedback, retry, cleanup, and
  DLQ worker roles.
- `PushFeedback`: provider response classification and token cleanup events.
- `PushMetrics`: counters, gauges, histograms, and labels for all stages.
- `PayloadTransformer`: template, localization, provider payload generation,
  payload size validation, and redaction hints.

## Credentials

Credentials must support:

- encryption at rest when stored by Sockudo
- versioning
- staged rotation
- active/previous credential overlap
- redaction in logs, metrics, errors, and debug endpoints
- in-memory cache invalidation on rotation or admin revocation
- provider-specific validation without exposing secret material

Credential cache keys must include app, provider, credential version, and
environment. Rotation must invalidate provider clients that depend on stale
credentials.

## Reliability Semantics

### Idempotency

Publish APIs must accept an idempotency key or derive one from request identity
where appropriate. Duplicate requests must return the original `publish_id` or
equivalent result when the semantic request matches.

### Scheduling, Cancellation, And Expiry

Scheduled records must be durable and cancellable before dispatch. Expired
publishes must not dispatch and must emit status/metrics.

### Retries And Dead Letters

Retry policy must include provider class, status, retry-after, exponential
backoff with jitter, max attempts, max elapsed time, and expiry. Exhausted jobs
must move to dead letters with enough context for safe replay.

### Duplicate Suppression

Duplicate suppression must cover request idempotency, fanout shard replay,
worker redelivery, provider retry, and cancellation races.

## Fairness And Backpressure

Release 4.5 must enforce:

- per-tenant publish acceptance ceilings
- per-provider rate limits
- per-app fanout caps
- weighted fair queueing across tenants
- noisy-tenant isolation
- dispatch backpressure when providers or queues degrade
- admission rejection or `202` degradation with explicit status when capacity is
  constrained

A tenant sending 10x quota must not degrade other tenants' p99 latency by more
than 20% in release candidate load tests.

## Provider Quota Realism

Sockudo can accept, persist, plan, queue, and dispatch at high throughput.
Sockudo cannot force FCM, APNs, Web Push push services, HMS, or WNS to exceed
their provider quotas, abuse limits, regional capacity, or endpoint policies.

Docs and metrics must distinguish:

- Sockudo publish acceptance capacity
- Sockudo fanout planning capacity
- Sockudo dispatch worker capacity
- provider accepted/acknowledged notifications
- provider-throttled notifications
- provider-deferred or failed delivery

No scale claim is supported unless backed by measured evidence in the target
environment.

## Security And Privacy

Binding rules:

- Provider tokens and Web Push auth secrets are secrets.
- Token hashes may be used for lookup and deduplication; raw tokens must not
  appear in logs, metrics, traces, or ordinary status responses.
- Payloads may contain user data and must be redacted from logs by default.
- Metrics labels must not include raw tokens, device IDs, user IDs, payload
  fields, or credential identifiers that reveal secret material.
- Admin APIs must be capability gated and auditable.
- Dry-run responses must not expose secrets.
- Dead letters must redact provider tokens and credentials while preserving
  enough metadata for replay by authorized operators.

## Verification And Release Blockers

Release 4.5 cannot ship until all blockers are satisfied:

- Complete device registry and channel push subscriptions.
- Async publish returns `202 + publish_id`.
- Cursor pagination on every list endpoint.
- Payload transformation layer implemented, not pass-through JSON.
- Scheduled delivery cancellable before dispatch.
- Per-device state tracked and observable.
- Stale device cleanup configurable.
- At least one real end-to-end provider verified before shipping.
- Tenant fairness and backpressure verified under load.
- No unsupported scale claim without performance evidence.

Required verification:

- Unit tests for payload transformation, idempotency, provider response
  classification, scheduling, cancellation, expiry, retries, duplicate
  suppression, credential redaction, and cursor pagination.
- Store conformance tests for every supported storage backend.
- Queue conformance tests for every supported queue backend and pipeline stage.
- Provider mock tests for FCM, APNs, Web Push, HMS, and WNS.
- At least one real provider end-to-end test in release candidate validation.
- Load tests for acceptance, fanout planning, dispatch, fairness, backpressure,
  and noisy-tenant isolation.
- Chaos tests for provider throttling, provider outage, queue redelivery, worker
  crash, storage failover, credential rotation, cancellation races, and stale
  cleanup.
- Soak tests covering sustained publish acceptance and dispatch with bounded
  memory, queue depth, store growth, and retry backlog.
- Documentation for API, configuration, feature flags, backends, operational
  modes, provider quota realism, migrations/provisioning, and rollout.

## Rollout Plan

Recommended release phases:

1. Merge architecture, release contract, dependency ADR, and API docs.
2. Add feature flags and empty `crates/sockudo-push` surfaces.
3. Implement memory store/queue/provider mocks and conformance tests.
4. Implement provider wrappers behind provider features.
5. Implement durable stores one backend family at a time.
6. Implement queue adapters and worker pipeline stages.
7. Add metrics, docs, migrations/provisioning, and load tests.
8. Run at least one real provider end-to-end validation.
9. Mark non-hyperscale backends as supported but not recommended for the
   largest deployment targets.
